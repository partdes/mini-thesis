# Article Structure Extractor

Extract crystal structure parameters from article text and output a YAML spec file for the `structure_builder` pipeline. Handles direct and non-straightforward descriptions.

**Usage:** paste article text after the command, or provide a file path like `data/articles/smith2023.txt`

## Procedure

### Step 1: Read input
- If a file path argument is given, read the file with the Read tool.
- If a DOI is given, fetch the abstract.
- Otherwise, use the pasted text.

### Step 2: Interpret non-straightforward descriptions
Before mapping to YAML fields, resolve indirect language:

| Article phrase | Interpretation |
|---|---|
| "64-atom rock-salt supercell" | `supercell_matrix: [4, 4, 4]` (2-atom primitive × 4³ = 128 sites → check atom count) |
| "N-atom supercell" | Divide by primitive cell atom count to find scaling. Flag for confirmation. |
| "equimolar" / "equiatomic" / "equal fractions" | Divide 1.0 equally among all named elements |
| "SQS" / "special quasi-random structure" | `type: sqs` |
| "following Zunger et al." / "ATAT" | `sqs_options.backend: auto` |
| "alternating (001) planes" | `arrangement: layered`, `layer_axis: 2` |
| "alternating (110) planes" | `arrangement: layered`, `layer_axis: 1` |
| "L12 ordering" | `arrangement: ordered`, `ordered_type: l12` |
| "B2 ordering" | `arrangement: ordered`, `ordered_type: b2` |
| "rock-salt structure" | `base_structure: mp-492` (TiN) or infer from composition |
| "wurtzite structure" | `base_structure: mp-661` (hexagonal AlN) or infer |
| "described in [ref X]" | `# MISSING: derived from reference [X] — locate that paper` |

Every interpreted value must include an inline comment:
```yaml
supercell_matrix: [4, 4, 4]  # interpreted: "64-atom supercell", 2-atom primitive → 4x4x4
```

### Step 3: Extract parameters and build YAML

Map all found information to the YAML spec schema. Every field that cannot be determined is marked `# MISSING: <reason>`.

### Step 4: Output

Output a YAML code block ready to save to `data/specs/`. Include:
- A brief note on what was interpreted vs. read directly.
- A list of any `# MISSING` fields the user must fill in.

### Step 5: Suggest a filename

Suggest a name based on composition and structure type, e.g. `TiAlN_2x2x2_sqs`.

## Output format

```yaml
# Generated by article-structure-extractor
# Source: <article title or filename>
# Interpreted fields are annotated with inline comments.
# Fields requiring manual input are marked # MISSING.

name: # MISSING: suggest e.g. TiAlCrHfN_4x4x4_sqs
base_structure: mp-492  # inferred: rock-salt TiN
type: sqs
supercell_matrix: [4, 4, 4]  # interpreted: "64-atom supercell"
composition:
  metal:
    Ti: 0.25
    Al: 0.25
    Cr: 0.25
    Hf: 0.25
  nonmetal:
    N: 1.0
output:
  formats: [vasp, cif]
  directory: data/structures/
source:
  article: "Smith et al., Acta Materialia, 2022"
  description: >
    "quoted sentence from article"
  doi: # MISSING
sqs_options:
  backend: auto
  iterations: 50000
```

## What this does NOT do

- Does not run the builder — spec production only.
- Does not guess ambiguous values — flags them with `# MISSING`.
- Does not modify existing spec files — always outputs a new block.

## After extraction

Save to `data/specs/<name>.yaml`, then run:
```powershell
python -m structure_builder data/specs/<name>.yaml
```
